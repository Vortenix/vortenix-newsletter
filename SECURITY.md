# Security policy

## Supported versions

Until the first stable release, security fixes are made on the latest `0.x` release line and `main`. Older snapshots are not supported.

| Version | Supported |
| --- | --- |
| Latest `0.x` | Yes |
| Older versions | No |

## Reporting a vulnerability

Do not open a public issue. Use GitHub's private vulnerability reporting feature for this repository. If that feature is unavailable, contact the repository owner privately through their GitHub profile and request a secure reporting channel; do not include exploit details in the initial public message.

Include the affected version, impact, reproduction steps, and any suggested mitigation. Avoid accessing data that is not yours and stop testing once the issue is demonstrated.

Maintainers aim to acknowledge a report within seven days, provide an initial assessment within fourteen days, and coordinate a fix and disclosure timeline with the reporter. These are targets, not contractual guarantees.

## Scope and disclosure

High-priority areas include approval bypasses, unintended email delivery, credential exposure, unsafe handling of fetched content, path traversal, and dependency vulnerabilities. Please allow a reasonable remediation period before disclosure. Credit is offered unless the reporter prefers anonymity.
